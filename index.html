How It's Made

This website represents the first major update to my personal website.

Over time, I found more and more stuff that needs improvement. Then one day I decide on a whim that it would be a good day for an overhaul.

While I don't need a personal website right now, websites need proper maintenance and care for from time to time.

Accessibility

Accessibility was a big topic in web development in 2019. Dominos and Beyonce were sued for websites that violated the American Disabilities Act (ADA).

Besides semantic elements, color contrast and a sane document structure, I added an on-site text-to-speech feature. This helps both users too lazy to read and users with visual impairment. You might think that the latter would already have their own text-to-speech software setup, but these software don't take context into consideration. It's not uncommon for words like "three.js" to be pronounced "Three <pause> JS" instead of "Three Dot JS". The audio files I provide have been manually vetted to ensure such flaws are taken care of. The audio files are generated using Google's Text-to-Speech API with its premium WaveNet voice model. Input is provided as Speech Synthesis Markup Language (SSML) which allows for the control of various elements of speech like rate, pitch, contour and more. This is an ongoing effort and not all texts on the site have text-to-speech yet.

Other than that, I also designed for color blindness. To ensure that my color scheme is color blind friendly, I use the "Let's Get Color Blind" extension for Firefox by Nullbrains. It simulates how people with different color blindness see my site. I then use my own judgment to tweak them as needed. It's not perfect but its a good step forward. The types of color blindness I test for are protanomaly (reduced red hue sensitivity), deuteranomaly (green), tritanomaly (blue).

Accessibility is not only limited to people with disabilities. There are around a billion people on Earth with Chinese as their first language. Being a Malaysian Chinese myself, I took the initiative to localize my website into my mother tongue. Localized strings are stored in a single master list similar to the approach used by Twine. A custom script turns function calls in EJS into bilingual HTML. A language switcher powered by pure CSS is added to the frontend. The localization effort is still ongoing.

Moving on, there's the matter of keyboard-only navigation. I admit I hadn't thought of this until much later. However, I've since made efforts to accommodate keyboard-only users. Focused elements are now clearly distinguished and the correct elements are focused after page navigation and modal toggles. The former was my biggest gripe when I tried browsing the web using only my keyboard. Losing track of the currently focused element caused me the most frustration by far.

Next, I added an option in the settings page for a reduced motion mode. It disables all non-essential animations. My site makes heavy use of animations and I understand that some users might not like that either due to personal preference or health reasons like motion sickness. There is a "prefers-reduced-motion" media query being considered for the Media Queries, Level 5 specification, but it's at the Editor's Draft stage. I'll wait have it to reach Working Draft status before considering supporting it. There's the text scaling slider too, a staple of government websites. This setting may seem pointless but don't forget that not all mobile browsers allow for text-only scaling and less computer-savvy users may be unfamiliar with browser settings.

Additionally, there is now support for devices with high refresh rate displays. Most developers assume the display refresh rate of their users to be 60Hz. As someone with a 144Hz monitor, this assumption makes certain time-sensitive web games unplayable for me. Do note that support for high refresh rate displays does not mean limiting everyone's framerate to 60fps, that's the lazy way out. Instead, in my implementation, value changes are scaled based on the time between updates. For example, if a bullet is expected to move 10px every 1/60 of a second, it will only move 5px if it has only been 1/120 of a second since the last update. As a result, my implementation takes advantage of higher framerates to deliver a smoother experience.

Finally, there are users on Internet Explorer (IE) and users without JavaScript. I know many developers snub their noses on these groups but I believe it's not our place to judge other people's circumstances or lifestyle choices. The previous version of my website was IE11 compatible, this version is now IE10 compatible except for the WebGL stuff. As for no script compatibility, you would still be able to navigate to all text content, even those in modal boxes. However, advanced acessibility feature will be crippled. This is not a deficiency on my part but a limitation of the current state of web accessibility. For example, even the WAI-ARIA Authoring Practices 1.1 example for modal dialogs uses JavaScript for controlling focus and setting ARIA attributes. I've only tested my website with JavaScript disabled on the browser level. I make no guarantees if you use an extension to block JavaScript. These 2 properties combined makes this website usable even in hardened browsing environments like IE10's high security configuration with enhanced protected mode enabled.

Design

For the colour scheme, I was in love with this piece of artwork by Noya at the time and felt that my website would look good with a similar colour palette. As for the typeface, my favorite one at the time was the Mini Serif typeface developed by KKLD for Mini. However, being a proprietary font, I can't use it for my website. Nevertheless, I find Roboto Slab to be a similar typeface but with the added benefit of being bold enough for my taste. Thus I made that my header font. I didn't like it as a body font though. I felt that it was too sparse and too angular. I experimented with a few more typefaces at Google Fonts before settling with Bree Serif. It has just the right amount of density and roundness with a small x-height which I compensated for by upping the font size. As for the Chinese typeface, there weren't a whole lot of good choices to go around so I just went with the only good option on Google Fonts: Noto Sans SC.

While the hero animation on my first website was technically impressive, it was quite GPU intensive. The use of WebGL particles affected performance especially on mobile where even my phone struggled to keep up a smooth and consistent framerate. Here's how the home page of my website looked like before the makeover:

The new hero animation makes use of CSS perspective and 3D transforms. It uses almost a quarter less GPU time per frame than my old hero animation. I gathered ideas by watching MMD music videos and anime openings on YouTube because their art style is just what I need: 2D art that can be done using either SVG or procedurally using CSS. After a bit of searching, I found a good concept that I could use as reference in the opening movie of the 2014 Nintendo 3DS game, "Persona Q: Shadow of the Labyrinth":

The new website also sees the introduction of a new theme switcher. Currently, besides the default theme, 4 other themes are available: Caramelldansen, Moonlight, Dark Cherry Mystery and Ya Soshla S Uma:

  • I'm sure most people who grew up through the 2000s would have heard Caramelldansen. It is one of the most recognizable tracks of that era alongside other classics such as "Axel F" by Crazy Frog, "Ievan Polkka" by Hatsune Miku and "Fabulous Secret Powers" by Slackcircus. It got stuck in my head again not long ago and I thought why not make a theme for it. The Web Audio API is used for the waveform and sprite animation is used for the dance. The animation loop I used is the one that spawned the Caramelldansen meme. I extracted the artwork from Timon1771's Caramelldansen flash video, cleaned it up and turned it into a sprite sheet.
  • The Moonlight theme was available as an alternate theme on my old website and I felt it looked too good not to be ported over. I created the vector art myself in Inkscape, using scenes from the original anime as reference. It is also the only theme to feature Dolby Digital encoded 5.1 surround sound audio.
  • The Dark Cherry Mystery theme can be unlocked by completing the zombie rail shooter game available as an Easter egg in the default theme on widescreen displays. It features a mysterious particle effect created using the Canvas API.
  • Before Hozier's "Take Me To Church" and Lady Gaga's "Born This Way", there was "Ya Soshla S Uma", or more widely known is its translated version "All the Things She Said". I was watching the music video for "D City Rock" by TeddyLoid and one of the scenes paid homage to this song. I felt that that scene looked really good, thus I tried to reproduce it in CSS for my website. I can't post a screenshot here because its a bit raunchy but feel free to google for it after work.

A 6th theme called Red Angels is currently being worked on. It will be based on the song "Red Angels" by James Shimoji, which is composed as an original soundtrack for the movie "Redline".

Regarding the layout, I stuck with the multipage layout of my old website. I felt that by cleanly separating each section, users would have a clearer visual map and have an easier time navigating around. Page transitions are now enhanced with JavaScript for increased smoothness and robustness with graceful degradation to the original pure CSS solution if JavaScript is disabled.

This update also sees responsive and fluid design taken to the next level. The entire website adapts seamlessly to the most extreme of resolutions and aspect ratios. It looks equally good regardless if it's viewed on a 4K Dell UltraSharp, a 368x448 Apple Watch Series 5 or even a 240x360 Nokia 8110 4G. Feel free to open up your browser's developer tools and stretch that viewport to see for yourself.

Build Tools

Other than the redesign, I also retooled my build tools. Instead of Webpack, I replaced it with Gulp. Webpack wasn't fast enough for me especially during hot-reloading. I didn't need its module bundling feature considering the small number of modules and shallow dependency tree that I have. For my needs, a simple task runner will do. Instead of raw CSS, I now use Sass with CSSComb integration for consistent formatting and rule order across files. ESLint is also used for linting JavaScript files. Besides that, instead of the HTML-Webpack-Plugin, I now have a proper view engine in EJS. EJS was recommended to me by my first ExpressJS tutorial and it's been my go-to templating engine ever since. As for how long ago that was, during that time Pug was still called Jade. Regrettably, I have yet to find a reliable linter for EJS files. Other than that, I added in some quality of life improvements like automated font-subsetting using FontMin and automated screenshot taking of external links using Puppeteer. In the future, I would also add in automated image optimization using TinyPNG CLI.

Privacy

The revamped version of my website sees enhanced accountability with the inclusion of a privacy policy. Considering the Guestbook feature of this website collects name and IP addresses, even if it's explicit, I felt that users have the right to know how their data is managed and handled. This website is a privacy safe zone, it does not and will never contain any tracking cookies, tracking pixels, fingerprinters or social media trackers (The social media buttons on the "Other" page use a link-based approach with the "noreferrer noopener" attribute set). If you want to share information about yourself, submit a Guestbook entry!

Pieretta

Link to Demo

I made a simple WebGL application that renders a 3D model in real-time. This project represents my first unguided attempt working with 3D objects in WebGL. My previous WebGL projects had only involved 2D shapes. The purpose of this project is to improve my WebGL and GLSL skills as well as explore various common rendering techniques.

In this project, the Phong reflection model is used for local illumination while normal mapping and parallax occlusion mapping are used to enhance surface detail. The supersampling method is used for anti-aliasing with downsampling being done using cubic B-spline interpolation. Unlike my other projects where there is an emphasis on making things palatable to old browsers on weak devices, this time around, the focus is on modern code and visual fidelity. This project is also my way of celebrating Hatsune Miku's 12th birthday on the 31st of August 2019. What better way to celebrate a virtual diva's birthday than to work on a virtual project together with her. Happy Birthday, Miku!

The 3D model used is the Hatsune Miku Pieretta version model featured in the game "Hatsune Miku: Project Diva X" for the song "Cat Food". The model is distributed in the PMX format. To have Miku pose for the camera, I changed the default pose of the model by baking pose data from a VPD file using PMXEditor. The pose I chose is similar to the one used on the official figurine by Sega. Next, I converted the model into a JSON file. At first, I tried making my own script for the job as a learning experience, but while it wasn't hard, it was awfully tedious. After a while, I decided to save time by using the MMD loader plugin for three.js to do the parsing and then modifying the output to conform to the structure I wanted. Since I don't intend to have my application perform runtime conversion of PMX files, it wasn't really that important how I obtain the JSON files.

With preparations out of the way, it was time to begin work on the application itself. First, I started with the bare minimum code that allowed me to display my model. Next, I added in transformation matrices for the camera and turntable. Then implemented the Phong reflection model and added in a warm directional light. The Phong reflection model is used to simulate the effects of light on the model and consists of 3 components (ambient, diffuse and specular lighting), However, I omitted the specular part as there wasn't anything shiny in the scene.

Next up, I decided to implement normal mapping and parallax occlusion mapping. The reason for this is that the model's pigtails look really flat as if someone wrapped them in wallpaper. Some shading magic would need to happen to make those bumps and crevices on her hair look more convincing.

Normal mapping is used to manipulate the lighting into behaving as if it encounters a rough surface even though the actual surface is smooth. A "normal" is a unit vector that tells you which direction the surface is facing. Without normal mapping, the normals of all fragments within a polygon are uniform and are interpolated from the normals of the polygon's vertices. With normal mapping, normals can be defined on a per-fragment basis.

Humans use parallax to gain depth perception. Place your fists in a straight line away from your eyes. Move your head to the left and you would see that the further fist is on the left of the closer fist, move the other way and its now on the right. This is the parallax effect and parallax mapping simulates this effect to give the illusion of depth by displacing the texture coordinates. Parallax occlusion mapping is an enhancement of parallax mapping that uses multiple samples and linear interpolation between depth layers for better results.

Normal and parallax occlusion mapping requires a normal map and displacement map respectively to work. Since the original model doesn't have them, I used CrazyBump to auto-generate them from the diffuse map. I only use these techniques on the hair because these are not set and forget techniques, careful tweaking is required to make them look good, doing it wrong can easily make things worse. Unfortunately, auto-generated maps are not always perfect or good and since I lack the skills to improve them, it's for the better that I only use them when they're really needed.

Moving on, while WebGL has a build-in multisampling anti-aliasing (MSAA) feature that developers can request for, browsers can choose to ignore that request for performance, compatibility or whatever reasons. On normal density displays, the lack of anti-aliasing will result in terrible renderings full of jagged edges. On high dpi displays, most of the time anti-aliasing is not needed if you do it right, for example: Use physical pixels rather than CSS pixels when setting the dimensions of the drawing buffer, make sure no implicit conversions from decimals to integer occurs when dealing with dimensions, etc. Not wanting to stake the quality of my work on the whims of the browser, I decided to disable the build-in anti-aliasing altogether and roll with an alternative solution for normal density displays.

At first, I tried fast approximate anti-aliasing (FXAA), which is described as a postprocessing effect that intelligently blurs edges. It's quite popular and while it is indeed fast, results were lackluster. Images were blurry and although aliasing effects were reduced, they were still noticeable. For my second attempt, I decided to go with 4x supersampling (SSAA). SSAA is one of the oldest and simplest tricks in the book. It involves rendering at a resolution higher than the display resolution and scaling it back down. 4x means the internal resolution is at 2x width and 2x height for a total of 4x surface area. Although SSAA is quite demanding and relatively resource inefficient, it does an exceptional job of improving image quality. While MSAA only tackles aliasing on polygon edges, SSAA not only affects both polygon edges and textures, it is also noticeably more effective.

Regarding the downsampling algorithm for my SSAA, cubic B-spline interpolation is used. So what is a B-spline? Let's just say: In order to produce cubic filters suitable for image construction, the 2 parameter cubic convolution algorithm described by Mitchell and Netravali in Reconstruction Filters in Computer Graphics is used. Those 2 parameters are given the name B and C. B-spline filters are cubic filters with B=1 and C=0. When cubic filters are applied in 2D space, we call them bicubic filters. Other popular cubic filters include Catmull-Rom (B=0, C=1/2) and Mitchell-Netravali (B=1/3, C=1/3). Coming back to the topic of my downsampling adventures, at first, I experimented with using only the hardware bilinear filtering, the results were decent but I felt it could be better, so I decided to explore bicubic filtering. Bicubic filtering requires 16 samples to work, but by studying Bicubic Filtering in Fewer Taps by Phill Djonov, the implementation I used only requires 4 texture reads need to be performed. The way this is done is by first having the GPU use hardware bilinear filtering to blend 4 samples into 1, then take the weighted average of 4 of those samples to obtain the final output. Phill Djonov also mentions a Catmull-Rom filter using 9 samples instead of 16, but when I tried that out, I wasn't able to see any improvements and it was 5% more expensive.

Performance-wise, I've been able to achieve a smooth framerate with the 3-year-old Qualcomm Snapdragon 625 SoC on a vertical 1080p screen. On a horizontal 1080p screen, the Intel UHD Graphics 630 GPU manages to achieve 60 fps with around 40% utilization and 144fps with around 75% utilization. On a 2560x1600 Retina display with an Intel Iris Graphics 6100, I'm also getting a solid 60fps. Bandwidth-wise, the entire webpage with the relevant resources gzipped excluding audio weights in at about 2.5MB.

The on-vocal audio is sourced from the "Hanataba ~ the Best of Doriko feat. Hatsune Miku" album, karaoke audio is sourced from Doriko's personal website. Loudness normalization was performed on both files using Audacity's implementation of the ReplayGain 1.0 specification.
